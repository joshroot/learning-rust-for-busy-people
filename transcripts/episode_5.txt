Hello everyone, and welcome to the "Learning Rust For Busy People" podcast! This podcast is for busy people who want to learn and quickly be productive with the Rust programming language.

My name is Josh and today I’ll focus on the third part of our discussion about the topic of extracting text from a Markdown file.

In the previous episode, I talked about version 2 of the practical code example for this topic in which we refactored our program into a binary crate and a library crate to adhere to the concept of separation of concerns, we learned how to pass command-line arguments to our program, and we discussed the concept of "ownership" in a Rust program.

One major limitation in version 2 of the code is that the program panics in three scenarios, which means that the user sees a lot of technical information printed to standard error. In one of these scenarios, the program panics because of a bug in the program that allows an attempt to use an out-of-bounds index to access an array element. In the other two scenarios, the program panics because the "expect" method is called (one scenario is for a "Result" enum, the other is for an "Option" enum). The "expect" method panics if the "Result" enum is the "Err" variant or if the "Option" enum is the "None" variant.

Today, I’ll discuss version 3 of the code which incorporates new approaches for error handling.

Before diving in, I want to mention some key points that are described in the book "The Rust Programming Language" in the chapter titled "Error Handling", as well as in the online Rust language documentation for the "panic!" macro. In contrast to the Python programming language, Rust does not have exceptions. Instead, Rust has the concept of "unrecoverable" errors and "recoverable" errors, so it’s more granular in the sense that an error is thought of as belonging to one of these two categories.

The critical difference between these two types of errors is that an "unrecoverable" error is caused by a bug, whereas a "recoverable" error is something that can be anticipated.

For an "unrecoverable" error, we want the program to "panic", which means that it terminates immediately. Essentially, there’s a problem that cannot be resolved, so the safe thing to do is to just stop program execution. We can call the "panic!" macro directly to make the program panic. Or alternatively, a given operation can cause the program to panic. A clear example of this was in the previous version of the practical code example, where a missing command-line argument caused an "index out of bounds" error because the vector of command-line arguments did not have as many elements as expected.

For a "recoverable" error, we’re dealing with problems at runtime that we can anticipate. In the previous version of the practical code example, there were two examples of problems that we could anticipate ahead of time, which were trying to read the contents of a file that doesn’t exist, and trying to read the first line of an empty file. The "expect" method was called in both of these scenarios, which basically meant that if there was a problem (either a non-existent file or an empty file), the program would panic. Since these are problems that we can anticipate, we can handle these more gracefully instead of just letting the program panic.

Let’s dive in to version 3 of the practical code example to see how we can put the idea of "unrecoverable" errors and "recoverable" errors into practice.

We’ll begin with the "lib.rs" file, which is the library crate. There’s a new function called "get_file_path" that takes one argument, which is a reference to the vector of command-line arguments. This function gets the path for the Markdown file from the vector of command-line arguments and then returns the file path as a reference to a string. In addition, it also checks for a minimum length of the vector of command-line arguments. If a command-line argument was not passed when running the program, then the vector is too short and the "panic!" macro is explicitly called to make the program panic. We declare this function to be public since we’ll be calling this from the binary crate.

Let’s read through the function definition, line by line. On the first line we have "pub", "fn", "get_file_path", and then in parentheses we have "args", followed by a colon, then an ampersand operator immediately followed by a set of square brackets. Inside the square brackets is the word "String" (with a capital "S"). Based on my understanding, this syntax signifies that the "args" parameter for this function is a reference to a vector of String values.

The "get_file_path" function has a return value, so after the parentheses we have an arrow, which is composed of a hyphen immediately followed by a right-pointing angle bracket. (Note that the Rust arrow syntax for a return type is similar to Python syntax for a return type.) Then we have an ampersand operator immediately followed by the word "String" (with a capital "S") to indicate that this function returns a reference to a String. At the end of the line, we have an open curly bracket to begin the function body.

Inside the function body, we have an "if" expression to check the length of the vector of command-line arguments, which reads "if", "args.len", immediately followed by an empty set of parentheses, the less than symbol, and then the number 2. At the end of the line we have an open curly bracket and then on the next line, indented by four SPACE characters, we have a single statement which is a call to the "panic!" macro. The message "Not enough command-line arguments" is passed to the "panic!" macro. The vector of command-line arguments should have a length of 2 (because the element at index 0 is the name of the program binary and the element at index 1 is the Markdown file path that’s passed on the command line when running the program). If the Markdown file path is not passed, then this vector has a length of 1 (which is less than 2), which means the condition in the "if" expression is met, so the program panics and displays the provided error message. This is an "unrecoverable" error because the program needs the path to a Markdown file. If we don’t have this file path, we need to terminate the program immediately, so panicking is the right approach.

If the Markdown file path is passed, then the vector of command-line arguments has the expected length, and the file path is obtained from the element at index 1 in the vector. This statement reads "let", "file_path", equals, then the ampersand operator immediately followed by "args" which is followed by the number 1 inside square brackets, and then a trailing semicolon.

The last line in the function body reads "file_path", with no trailing semicolon. Since there is no trailing semicolon, this is an expression, not a statement. The convention in Rust for a function with a return value is to have the return value be the final expression in the function body. The "return" keyword exists in Rust and can be used to return early from a function. However, the "return" keyword is not needed if the return value is the final expression in the function body. So basically, you can just put the return value as the last line in the function body, and leave out the trailing semicolon.

Let’s move on and talk through the changes in the "run" function in the library crate. At a high level, there are two key changes to this function. The first change is that this function now has a "Result" enum as its return type. The second change is that the "expect" method is no longer called on the "Result" enum and "Option" enum in the body of the "run" function.

Let’s start with the return type. On the line that begins with "pub", "fn", "run", there is now an arrow after the parentheses that contain the function parameters. After the arrow, we have the word "Result" (with a capital "R"), which is immediately followed by a set of angle brackets.  There are two items inside the angle brackets, and they are separated by a comma. These two items correspond to the types for the "Ok" variant and the "Err" variant of the "Result" enum.

The first item inside the angle brackets is a set of empty parentheses, which is referred to as the "unit" type. In the online documentation for the Rust standard library, it says the "unit" type, and I quote, "is used when there is no other meaningful value that could be returned", end quote. The documentation indicates that functions without an explicit return type will implicitly have the "unit" type as its return type. This last point is important because this means that previously, the "run" function in the library crate had the "unit" type as its return type because a return type was not explicitly provided.

The second item inside the angle brackets begins with the word "Box" , which is immediately followed by a set of angle brackets that contain the word "dyn" (all lowercase), then a SPACE, then the word "Error" (with a capital "E"). "dyn" is a keyword that stands for "dynamic". "Error" is a "trait". Honestly, I don’t yet fully understand the concept of "traits" in Rust, so I’ll wait and talk about this concept after I figure it out myself. The "Error" trait is brought into scope via a new "use" statement at the top of the library crate, which reads "use", then "std", two colons, "error" (all lowercase), two colons, then "Error" (with a capital "E").

That’s a lot of new syntax in the return type specified for the "run" function, so I’ll give a high-level summary as I understand it. Basically, this return type syntax means the "run" function returns a "Result" enum  that will either be the "Ok" variant (that contains the "unit" type) or the "Err" variant (that contains an error value which can be of different types, hence the presence of the keyword for "dynamic").

Now let’s move inside the body of the "run" function and talk about what we’re using instead of the "expect" method calls. The "read_to_string" function returns a "Result" enum, and previously we called the "expect" method immediately afterwards. Now, we’ve removed the "expect" method call and replaced it with a single question mark, which is called the question mark operator.

Use of the question mark operator leads to the following behavior. If the "Result" enum that precedes the question mark operator is the "Ok" variant, the value contained by the "Ok" variant is returned from the expression. If the "Result" enum is the "Err" variant, the entire function (in this case, that means the "run" function) stops early (or we could also say the function returns early) and it returns the error value. This error value is then passed on to the code that called the "run" function. So, in short, the question mark operator enables us to propagate an error value to the calling code.

This changes the behavior of our program. Previously, if the "read_to_string" function returned a "Result" enum that was the "Err" variant, the program would panic and display the message that was provided to the "expect" method. That means we were treating this like an "unrecoverable" error since we were allowing the program to panic.

Now, we’re treating this as a "recoverable" error since we’ve anticipated the possibility of failure ahead of time and we’re doing something different than letting the program panic. We’re instead propagating the error value to the calling code via a "Result" enum. In this case, the "calling code" is the "main" function in the binary crate since that’s where the "run" function is called. By propagating the error value to the "main" function, we’re indicating that we want the "main" function to be responsible for deciding how to handle the error. This might mean attempting to recover from the error, or it might mean calling the "panic!" macro.

The book "The Rust Programming Language" indicates that if it’s possible a function will fail (which I think means if there’s a "Result" enum used somewhere in the function body), then it’s good practice to have the function return a "Result" enum. Another way I think about this is that if we’re defining a function that is not the "main" function, we shouldn’t assume that we have enough knowledge to know how to respond to an error. The best thing is to just propagate the error to the calling code. The book also indicates that one of the responsibilities of the "main" function is to handle errors from the "run" function. So, I interpret that as meaning that we want to propagate error values up to the "main" function, and then let the "main" function decide how to handle them. I’m sure there is some nuance to this, so I’ll touch on this topic again in the future after I have a more complete understanding.

There’s one other place in the body of the "run" function where we can use the question mark operator, and that’s the second time we used the "expect" method. This was when we were getting the first line in the Markdown file by creating an iterator (to iterate over each line in the   String with the file contents) and then calling the "next" method to get the very first line. The "next" method returns an "Option" enum, which is either the "Some" variant (if the first item in the iterator exists) or the "None" variant (if the first item in the iterator is absent or invalid). Previously, we called the "expect" method on the "Option" enum, which was basically a quick-and-dirty way to get the value we care about from the "Some" variant. I say "quick-and-dirty" because the downside of this approach is that if the "Option" enum is the "None" variant then the "expect" method causes the program to panic. This means we were treating this like an "unrecoverable" error since we were allowing the program to panic. However, we can anticipate ahead of time that there might be an issue when reading the first line in a file’s contents, so in my opinion this is more appropriately handled as a "recoverable" error.

It turns out that we can also use the question mark operator on an "Option" enum. Use of the question mark operator on an "Option" enum leads to the following behavior. If the "Option" enum that precedes the question mark operator is the "Some" variant, the value contained by the "Some" variant is returned from the expression. If the "Option" enum is the "None" variant, the entire function (again, in this case, that means the "run" function) returns early and it returns None.

The enum on which the question mark operator is used needs to match the return value of the function. If the question mark operator is used on a "Result" enum, then the function that contains this operation must return a "Result" enum. If the question mark operator is used on an "Option" enum, then the function that contains this operation must return an "Option" enum.

This presents us with a problem because inside the "run" function, the first time we use the question mark operator is for a "Result" enum and the second time we use it is for an "Option" enum. We just defined the return type of the "run" function to be a "Result" enum. This means we need to commit to dealing with either a "Result" enum or an "Option" enum. Since we’re dealing with error handling, a "Result" enum seems most appropriate.

So, we need to transform the "Option" enum to a "Result" enum. We can do this by calling the "ok_or" method on the "Option" enum. The "ok_or" method transforms an "Option" enum into a "Result" enum by mapping the "Some" variant to the "Ok" variant (while retaining the contained value) and by mapping the "None" variant to the "Err" variant. A custom error value is passed to the "ok_or" method to serve as the error value contained by the "Err" variant.

To implement this in our code, we go to the line where the "next" method is called, we remove the call to the "expect" method, then we add ".", "ok_or", parentheses that contain the message "File is empty", then the question mark operator, then a trailing semicolon.

There’s one final change inside the body of the "run" function, and that’s the addition of one new line of code at the very end. This line reads "Ok", immediately followed by a set of parentheses, and inside the parentheses is a set of empty parentheses, which is the "unit" type. There is no trailing semicolon. This means that this line is an expression and is the value returned by the function. So, if all operations in the body of the "run" function complete successfully (in other words, no operations fail), the function will return a "Result" enum that is an "Ok" variant that contains the "unit type". This corresponds to the return type that we specified for the "run" function.

Now that we’ve discussed all the changes inside the body of the "run" function, let’s summarize it’s behavior at a high level. If all operations inside the "run" function complete successfully, nothing of value is returned from the function. This makes sense because inside the "run" function, the final operation of interest for us is to simply print the extracted title of the Markdown file to standard output. There’s nothing that the "run" function needs to return. If, however, one of the operations in the "run" function fails (which means that either the "read_to_string" function fails or trying to get the first line of the file contents fails), then  this failure triggers an early return of the function and the error value is propagated to the calling code.

As I mentioned earlier, the "calling code" is the "main" function in the binary crate since that’s where the "run" function is called. Let’s shift our focus to the binary crate to talk through the changes in that file.

At the top of the "main.rs" file, which is the binary crate, we bring the "process" module (from the standard library) into scope by adding a "use" statement. This reads "use", then "std", two colons, "process", and a semicolon. We’ll talk about how we use the "process" module shortly.

Inside the body of the "main" function in the binary crate, there are now two statements and one "if" expression. We’ve removed the calls to the "dbg!" macro. The first statement is where the "args" variable is defined to be a vector of command-line arguments, and this is the same as before.

The second statement defines a variable named "file_path" that is set equal to the value returned from the "get_file_path" function that’s defined in the library crate. This line reads "let", "file_path", equals, then the name of the package (which is listed in the "Cargo.toml" file) which is immediately followed by two colons and then the function name "get_file_path". Right after "get_file_path" we have a set of parentheses that contain an ampersand operator in front of the variable name "args".  Here, we’re passing a reference to the value owned by the variable named "args" to the "get_file_path" function. At the end of the line we have a trailing semicolon since this is a statement.

Now we arrive at the final step inside the "main" function, which is to call the "run" function in the library crate and to handle any resulting error value that was propagated. The "run" function doesn’t return any meaningful value via the "Ok" variant of the "Result" enum, so we really only care about handling the scenario when the "run" function returns a "Result" enum that is the "Err" variant.

To do this, we use what’s called the "if let" syntax to match against the "Err" variant of the "Result" enum. The "if" expression reads "if", "let", "Err" (with a capital "E"), immediately followed by a set of parentheses that contain a lowercase letter "e", equals, then the name of the package (which is listed in the "Cargo.toml" file) which is immediately followed by two colons and then the function name "run". Right after "run" we have a set of parentheses that contain the variable name "file_path". Then we have an opening curly bracket.

Quick side note – the value contained inside the "Err" variant is bound to the variable "e" that’s in parentheses right after the word "Err". The value of this variable "e" is accessible in the body of the "if let" block.

On the first line in the body of the "if let" block, indented by four SPACE characters, we have a call to the "eprintln!" macro, which is similar to the "println!" macro except that this prints to standard error instead of to standard output. Immediately after the macro name "eprintln!", we have a set of parentheses that contain a custom error message inside double quotes that says "Application error", colon, and then a lowercase letter "e" inside curly brackets. The variable "e" inside these curly brackets is the error value contained inside the "Err" variant.

This means that if an error value in the "run" function is propagated to  the "main" function, then the message printed to standard error will read  "Application error", colon, then whatever the value is for the corresponding error. We’ll see a couple examples of this when we compile and run the program.

The second line in the body of the "if let" block reads "process", immediately followed by two colons, then the function name "exit", which is immediately followed by a set of parentheses that contain the number 1. The "exit" function (from the "process" module that we brought into scope earlier) is used to immediately stop program execution and to return the provided value as the exit status code. At the end of this line we have a semicolon.

The next line contains a closing curly bracket which is outdented one level so that it’s aligned with the word "if", which ends the "if let" expression.

Now, we can compile and run our program for a few different scenarios.

Let’s open a command-line shell and run the command "cargo run -- ", and then the path to the example Markdown file. After running this command, we see a multi-line message that includes the words "Compiling", "Finished", and "Running", followed by the extracted title of the Markdown file.

If we run the same command and then redirect standard output and standard error to separate files, we’ll see that the standard output file has the extracted title of the Markdown file, as expected.

Now, we’ll go through the same three scenarios that previously caused our program to panic.

First, we’ll run the command "cargo run" without any arguments. We see a message that includes the word "panicked" and the message "Not enough command-line arguments". The program panicked because we explicitly call the "panic!" macro when there is an insufficient number of command-line arguments. This is an "unrecoverable" error.

Next, we’ll run the program for a Markdown file that does not exist. When we run the command "cargo run -- does_not_exist.md", the program no longer panics because we’re now treating this problem as a "recoverable" error. Now, instead of panicking, the program prints the following message to standard error: "Application error: No such file or directory".

And finally, we’ll run the program for a Markdown file that is completely empty by running the command "cargo run -- ", and then the path to the empty Markdown file. As with the previous scenario, the program no longer panics because we’re now treating this problem as a "recoverable" error. Instead, it prints the following message to standard error: "Application error: File is empty".

In my opinion, the two "recoverable" errors are now handled more gracefully because there is less technical information displayed to the user (compared to the information that is displayed when the program panics).

And with that, we’ve finished talking through the third version of our practical code example for the topic of extracting text from a Markdown file. To recap, we talked about how to implement error handling by treating problems as "unrecoverable" errors and calling the "panic!" macro, or by treating problems as "recoverable" errors and returning a "Result" enum. We also talked about how to propagate errors using the question mark operator.

I have two closing comments to wrap up this episode.

The first is that using the "expect" method or the question mark operator for a "Result" enum is not the only way to handle "Result" enums. There are other methods available and I simply chose an approach that seems robust and that seems aligned with the approach used in the "minigrep" code described in the book in the chapter titled "An I/O Project: Building a Command Line Program". The book contains detailed descriptions of other approaches for working with "Result" enums.

My second comment is that the book describes how the "expect" method is useful for prototyping. I’ve found it helpful to view the "expect" method as a quick-and-dirty approach for getting the value we want from an enum and then just panicking if something fails. When the focus shifts from prototyping to making the program more robust, the calls to the "expect" method are like placeholders that indicate where robust error handling eventually needs to be implemented.
